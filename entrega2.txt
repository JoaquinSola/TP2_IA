1. IMPLEMENTACIÓN BASE DEL AGENTE
══════════════════════════════════════════════════════════════════════

El agente es un sistema conversacional multimodal que asiste a personas con
discapacidad visual en el pago de facturas de servicios. Permite al usuario
fotografiar una factura, fotografiar los billetes disponibles, y recibe como
respuesta hablada qué billetes entregar y cuánto vuelto esperar.

Stack tecnológico implementado:
  - LLM de texto:  Llama 3.3 70B Versatile (via Groq API)
  - LLM de visión: Llama 4 Scout 17B 16E Instruct (via Groq API, multimodal)
  - STT (voz):     Whisper Large v3 Turbo (via Groq API)
  - TTS (síntesis): gTTS (Google Text-to-Speech, acento es-AR)
  - Orquestador:   LangGraph (máquina de estados con nodos especializados)
  - Backend:       FastAPI (Python 3.11+)
  - Frontend:      Vanilla JS, mobile-first, Web Speech API para desktop
  - RAG:           BM25Okapi (rank_bm25) sobre knowledge base estática
  - Logging:       JSON Lines estructurado (archivo .jsonl por sesión)

Estructura del proyecto:
  backend/
    main.py              → FastAPI, endpoints REST
    agent/
      graph.py           → Grafo LangGraph, nodos, enrutamiento
      state.py           → AgentState (TypedDict)
      prompts.py         → System prompt, decision prompt, formateadores
    tools/
      invoice_extractor.py   → extraer_datos_factura (LLM visión)
      bill_identifier.py     → identificar_billetes (LLM visión)
      payment_calculator.py  → calcular_cambio_y_pago (Python puro)
    rag/
      knowledge_base.py  → Corpus BM25, retrieve_context()
    models/
      schemas.py         → Pydantic: Bill, InvoiceData, PaymentResult, ChatRequest/Response
    sessions/
      manager.py         → SessionState en memoria (dict en proceso)
    observability/
      logger.py          → log_llm_call, log_tool_call, log_tool_result, log_error
  frontend/
    index.html           → UI accesible, mobile-first
    css/style.css
    js/app.js            → Lógica de voz, captura de imágenes, llamadas a la API

2. FLUJO DE INTERACCIÓN CON EL LLM
══════════════════════════════════════════════════════════════════════

El agente usa DOS modelos de Groq en roles distintos:

2.1 Modelo de texto (Llama 3.3 70B) — Nodo "agent"
  Responsabilidad: decidir la próxima acción del agente.
  Recibe un prompt de decisión con el estado actual (¿hay factura? ¿hay billetes?
  ¿hay imágenes pendientes?) y el mensaje del usuario. Responde con:
    - "TOOL:extraer_datos_factura"  → si hay imagen de factura sin procesar
    - "TOOL:identificar_billetes"  → si hay imagen de billetes sin procesar
    - "TOOL:calcular_cambio_y_pago"→ si tiene factura Y billetes
    - Texto libre                  → respuesta conversacional directa

  Parámetros de llamada:
    temperature=0.3, max_tokens=1024
    System prompt: identidad del asistente, reglas de moneda, tono argentino

2.2 Modelo de visión (Llama 4 Scout, multimodal) — Nodos de tools
  Usado por extraer_datos_factura e identificar_billetes.
  Recibe la imagen en base64 embebida en el mensaje junto con un prompt
  especializado. Responde siempre en JSON puro.

  Para facturas:
    temperature=0.1, max_tokens=1024
    Extrae: entity, total_amount, due_date, second_due_date, second_amount,
            is_valid_document, error_message

  Para billetes:
    temperature=0.1, max_tokens=1024
    Extrae: lista de objetos {denomination, position, valid, currency, confidence}

2.3 Modelo STT (Whisper Large v3 Turbo) — Endpoint /api/transcribe
  Recibe un archivo de audio (mp3/mp4/wav/ogg/webm).
  Retorna transcripción en texto plano en español.
  Usado desde mobile donde SpeechRecognition no funciona sobre HTTP.

Manejo de imágenes antes de enviar al LLM:
  Las imágenes se redimensionan si superan 3 MB (máximo lado: 1920px, JPEG 85%).
  Esto evita errores por límite de tokens de la API de visión.

3. MÓDULO DE CONTEXTO Y MANEJO DEL AMBIENTE
══════════════════════════════════════════════════════════════════════

3.1 Estado del agente (AgentState)
  Implementado como TypedDict en backend/agent/state.py.
  LangGraph lo propaga a través de los nodos; cada nodo retorna un dict
  parcial que se fusiona con el estado anterior.

  Campos del estado:
    messages            → lista de mensajes (HumanMessage / AIMessage),
                          manejada con add_messages (append automático)
    invoice_data        → dict con los datos de la factura (None hasta extraer)
    bills_data          → lista de dicts de billetes (None hasta identificar)
    payment_result      → dict con resultado del cálculo
    invoice_image_bytes → bytes de la imagen de factura (se limpia al procesar)
    invoice_image_mime  → MIME type de la imagen de factura
    bills_image_bytes   → bytes de la imagen de billetes
    bills_image_mime    → MIME type de la imagen de billetes
    next_action         → decisión del nodo agent ("extraer..." / "respond" / etc.)
    session_id          → UUID de la sesión del usuario
    rag_context         → string con los documentos recuperados por BM25
    final_response      → texto que el agente devuelve al usuario
    step                → nombre del último nodo ejecutado (para trazabilidad)

3.2 Gestión de sesiones (SessionState)
  Implementado en backend/sessions/manager.py como dataclass en memoria
  (dict Python por proceso). Almacena por session_id:
    - conversation_history  → lista de dicts {role, content} para historial
    - invoice_data          → datos de factura entre turnos
    - bills_data            → datos de billetes entre turnos
    - payment_result        → resultado del cálculo entre turnos
    - current_invoice_bytes → imagen de factura recibida en el turno actual
    - current_bills_bytes   → imagen de billetes recibida en el turno actual
    - awaiting              → estado de espera del flujo (None por ahora)

  El endpoint /api/reset limpia los datos de la transacción actual
  sin borrar el historial conversacional, permitiendo pagar otra factura.

3.3 Configuración del entorno
  Variables de entorno (archivo .env en raíz del proyecto):
    GROQ_API_KEY          → clave de la API de Groq (obligatoria)
    GROQ_TEXT_MODEL       → modelo de texto (default: llama-3.3-70b-versatile)
    GROQ_VISION_MODEL     → modelo de visión (default: meta-llama/llama-4-scout-17b-16e-instruct)
    PORT                  → puerto del servidor (default: 8000)
    LOG_DIR               → directorio de logs (default: logs/)

  La carga del .env se hace con python-dotenv desde una ruta absoluta
  (Path(__file__).parent.parent / ".env") para evitar problemas de
  directorio de trabajo al lanzar el servidor.

4. CONEXIONES EXTERNAS NECESARIAS
══════════════════════════════════════════════════════════════════════

4.1 API Groq (groq Python SDK)
  URL base: https://api.groq.com
  Autenticación: GROQ_API_KEY en header Authorization: Bearer.
  Endpoints usados:
    POST /openai/v1/chat/completions  → texto (Llama 3.3 70B)
    POST /openai/v1/chat/completions  → visión (Llama 4 Scout, con image_url)
    POST /openai/v1/audio/transcriptions → STT (Whisper Large v3 Turbo)

  El cliente Groq se instancia una sola vez (singleton _client en graph.py)
  y se reutiliza en todos los nodos mediante get_client().

4.2 gTTS (Google Text-to-Speech)
  Librería Python que llama a la API pública de Google Translate TTS.
  Genera audio MP3 del texto en español argentino (lang='es', tld='com.ar').
  Se usa en el endpoint /api/tts del backend para devolver audio al frontend.
  Mobile reproduce el MP3 con la Web Audio API.

4.3 No hay base de datos externa
  El estado de sesión vive en memoria del proceso FastAPI. Esto es suficiente
  para el alcance del TP (sesiones de una sola transacción). No se usa Redis,
  SQLite ni ningún motor externo.

4.4 Servicio de archivos estáticos
  FastAPI sirve el frontend (index.html, CSS, JS) directamente desde
  /static usando StaticFiles de Starlette, sin servidor web separado.

5. WORKFLOW
══════════════════════════════════════════════════════════════════════

El workflow es un grafo dirigido LangGraph con 6 nodos:

  [ENTRADA del usuario]
        ↓
  ┌─────────┐
  │   RAG   │  Recupera documentos relevantes de la knowledge base (BM25)
  └────┬────┘  y los agrega al estado como rag_context.
       ↓
  ┌─────────┐
  │  AGENT  │  El LLM de texto analiza el estado + mensaje + RAG context
  └────┬────┘  y decide la próxima acción (TOOL:xxx o respuesta directa).
       ↓ (enrutamiento condicional)
       ├──→ [extract_invoice]  Si hay imagen de factura pendiente
       ├──→ [identify_bills]   Si hay imagen de billetes pendiente
       ├──→ [calculate]        Si tiene factura Y billetes
       └──→ [respond]          Si es conversación directa
       ↓
  ┌──────────────────┐
  │  TOOL NODE       │  Ejecuta la herramienta correspondiente.
  │  (uno de los 3)  │  identify_bills puede encadenar directo a calculate
  └────────┬─────────┘  si ya existe invoice_data en el estado.
           ↓
  ┌─────────────┐
  │   RESPOND   │  Sanitiza el texto (elimina $ del output del LLM).
  └──────┬──────┘  Devuelve final_response al caller.
         ↓
       [END]

Flujo nominal completo (happy path):
  1. Usuario saluda → RAG → AGENT → RESPOND  (saludo de bienvenida)
  2. Usuario sube foto de factura → RAG → AGENT → extract_invoice → RESPOND
     Respuesta: "Factura de EPE por 22.000 pesos con vencimiento el 15/07/2025."
  3. Usuario sube foto de billetes → RAG → AGENT → identify_bills → calculate → RESPOND
     Respuesta: "Detecté 2 billetes: uno de 10.000 pesos a la izquierda y uno de 20.000 pesos
     a la derecha. Entregá el billete de 10.000 pesos a la izquierda y el de 20.000 pesos a
     la derecha. Tu vuelto debe ser de 8.000 pesos."

6. INTEGRACIÓN DE HERRAMIENTAS (TOOLS) Y FUENTES EXTERNAS (RAG)
══════════════════════════════════════════════════════════════════════

6.1 Tool 1: extraer_datos_factura
  Archivo: backend/tools/invoice_extractor.py
  Tipo: LLM multimodal (Groq Llama 4 Scout)
  Input:  imagen en bytes + MIME type + session_id + cliente Groq
  Output: objeto InvoiceData (Pydantic)
    { entity, total_amount, due_date, second_due_date, second_amount,
      is_valid_document, error_message }
  Lógica:
    1. Si no hay imagen → retorna InvoiceData inválido con mensaje de error.
    2. Consulta BM25 para enriquecer el prompt con contexto del proveedor.
    3. Redimensiona la imagen si supera 3 MB.
    4. Llama al LLM de visión con temperature=0.1 (mínima creatividad).
    5. Parsea la respuesta como JSON (con fallback a regex para limpiar markdown).
    6. Normaliza montos argentinos: "22.000,50" → 22000.50 (float).
    7. Loguea la llamada al LLM y el resultado.

6.2 Tool 2: identificar_billetes
  Archivo: backend/tools/bill_identifier.py
  Tipo: LLM multimodal (Groq Llama 4 Scout)
  Input:  imagen en bytes + MIME type + session_id + cliente Groq
  Output: lista de objetos Bill (Pydantic)
    [{ denomination, position, valid, currency, confidence }, ...]
  Lógica:
    1. Llama al LLM de visión con prompt especializado en billetes ARS.
    2. Parsea la lista JSON de billetes.
    3. Valida que denomination esté en denominaciones vigentes de Argentina.
    4. Marca valid=False para billetes desconocidos o moneda extranjera.
    5. Asigna posición espacial (izquierda, centro, derecha, arriba, abajo)
       para que el usuario sepa cuál tomar sin ver.
    6. Loguea la llamada al LLM y el resultado.

6.3 Tool 3: calcular_cambio_y_pago
  Archivo: backend/tools/payment_calculator.py
  Tipo: Python puro determinístico (SIN LLM)
  Input:  InvoiceData + lista de Bill + session_id
  Output: PaymentResult (Pydantic)
    { total_available, total_required, sufficient, bills_to_use,
      change, bills_to_keep, missing_amount }
  Algoritmo:
    - Si total_available < monto_factura → retorna sufficient=False + missing_amount.
    - Si alcanza → enumera todos los 2^n subsets de billetes (hasta n=20).
    - Elige el subset que cubre el monto con el MÍNIMO vuelto posible.
    - No usa greedy porque puede generar combinaciones subóptimas.
    - Retorna qué billetes entregar y cuáles guardar.
  Decisión de diseño: usar código Python determinístico (y no el LLM) elimina
  el riesgo de alucinaciones aritméticas, que en un sistema de pagos sería
  un error crítico.

6.4 RAG — Recuperación por BM25
  Archivo: backend/rag/knowledge_base.py
  Motor: BM25Okapi (librería rank_bm25)
  Corpus: 21 documentos estáticos en memoria sobre:
    - 10 proveedores de servicios argentinos (EPE, ASSA, Naturgy, Telecom, etc.)
    - 9 billetes ARS en circulación (1.000 a 200.000 pesos, descripción visual)
    - 2 documentos generales sobre estructura de facturas argentinas
  Funcionamiento:
    - Se tokeniza el corpus en memoria al iniciar la aplicación.
    - retrieve_context(query, top_k=3) retorna los 3 documentos más relevantes.
    - Se inyecta en dos momentos: en el nodo RAG (para el agente decisor)
      y dentro de extraer_datos_factura (para el extractor de facturas).
  Ejemplo: si el usuario pregunta por "EPE", el BM25 rankea primero el documento
  sobre EPE, que el LLM usa para entender el formato de esa factura específica.

7. LOOP / ESTRATEGIA DE RAZONAMIENTO DEL AGENTE
══════════════════════════════════════════════════════════════════════

El agente implementa una variante del patrón ReAct (Reason + Act) adaptada
a un dominio cerrado con tres herramientas fijas.

Estrategia:
  A diferencia del ReAct clásico (tool calling abierto e iterativo),
  el agente razona una sola vez por turno y luego actúa. No hay iteraciones
  internas dentro de un mismo mensaje del usuario. Cada turno sigue:
    1. RAG → recuperar contexto
    2. AGENT (razonamiento) → decidir próxima acción
    3. TOOL (acción, si corresponde) → ejecutar herramienta
    4. RESPOND → devolver resultado al usuario

Decisión de acción (nodo agent):
  El LLM recibe el estado codificado en lenguaje natural (variables booleanas
  simples como "Tiene factura: Sí / No") y el historial de los últimos 10
  mensajes. Responde con una de 4 opciones posibles, sin JSON intermedio:
    - TOOL:extraer_datos_factura
    - TOOL:identificar_billetes
    - TOOL:calcular_cambio_y_pago
    - Texto libre de respuesta

  Las reglas de decisión están explícitas en el prompt (prioridad 1-4),
  lo que convierte al LLM en un router simple para los casos con herramientas.
  El LLM solo genera texto libre cuando no hay herramienta aplicable.

Encadenamiento automático identify_bills → calculate:
  Si el usuario envía billetes y ya existe invoice_data en el estado, el nodo
  identify_bills_node encadena automáticamente hacia calculate (sin pasar por
  el nodo agent de nuevo). Esto minimiza llamadas al LLM y latencia.

Manejo del estado entre turnos:
  El grafo LangGraph es stateless por sí solo (se reinvoca desde cero en cada
  turno). El estado persistente entre mensajes vive en SessionState (memoria
  del proceso). Al inicio de cada turno, main.py reconstituye el AgentState
  desde la sesión.

Por qué no se usó LangChain tool calling estándar:
  El dominio es cerrado (exactamente 3 herramientas, con condiciones de activación
  deterministas). El patrón de routing explícito en el prompt es más predecible
  y fácil de debuggear que el tool calling automático para este caso de uso.

8. LOGGING BÁSICO DE LLAMADAS AL LLM Y TOOLS
══════════════════════════════════════════════════════════════════════

Archivo: backend/observability/logger.py
Formato: JSON Lines (.jsonl) — un evento por línea, un archivo por sesión del servidor.
Ubicación: directorio logs/ (configurable con LOG_DIR en .env)

Eventos registrados:

  session_start
    { timestamp, event_type: "session_start", session_id }

  llm_call  (registrado en cada llamada al LLM de texto o visión)
    { timestamp, event_type: "llm_call", session_id, node,
      prompt_summary (primeros 300 chars), response_summary (primeros 200 chars),
      latency_ms, tokens_used }

  tool_call  (registrado antes de ejecutar cada herramienta)
    { timestamp, event_type: "tool_call", session_id, tool,
      inputs: { mime, size_bytes } o { invoice_amount, bills_count } }

  tool_result  (registrado después de ejecutar cada herramienta)
    { timestamp, event_type: "tool_result", session_id, tool,
      result: { ...datos extraídos... }, latency_ms }

  error
    { timestamp, event_type: "error", session_id, node, error: "mensaje" }

Medición de latencia:
  Se implementó un context manager Timer (clase en logger.py) que mide el
  tiempo de ejecución de cada llamada al LLM en milisegundos:
    with Timer() as t:
        response = client.chat.completions.create(...)
    log_llm_call(..., latency_ms=t.elapsed_ms)

Acceso a logs en tiempo real:
  GET /api/logs/{session_id} devuelve todos los eventos de una sesión en JSON.
  Esto permite auditar qué llamadas hizo el agente, con qué inputs y en cuánto
  tiempo respondió cada modelo.

Ejemplo de log de una sesión completa (extracto):
  {"timestamp":"2026-06-15T14:30:01","event_type":"session_start","session_id":"abc123"}
  {"timestamp":"2026-06-15T14:30:03","event_type":"llm_call","node":"agent_node",
   "latency_ms":412.5,"tokens_used":843}
  {"timestamp":"2026-06-15T14:30:03","event_type":"tool_call","tool":"extraer_datos_factura",
   "inputs":{"mime":"image/jpeg","size_bytes":245120}}
  {"timestamp":"2026-06-15T14:30:06","event_type":"llm_call","node":"extraer_datos_factura",
   "latency_ms":2847.3,"tokens_used":1102}
  {"timestamp":"2026-06-15T14:30:06","event_type":"tool_result","tool":"extraer_datos_factura",
   "result":{"entity":"EPE","total_amount":22000.0,"due_date":"15/07/2025","is_valid_document":true},
   "latency_ms":2848.1}
